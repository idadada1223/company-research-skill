最終成果物は **Markdown 1 枚**（ES 作成・面接対策を意識した構造）。
出力先のフォルダは"C:\Users\takaakiida\obsidian\ObsidianVault\.raw"とするが，ユーザに確認をとる．

## 実行フロー

### Step 0. 前提（ユーザー固定情報）
以下はユーザー固有の前提として**固定**。毎回確認しない。
- **志望年度：28 卒**（選考情報は 28 卒向け最新情報を収集）
- **志望職種：IT 技術系（広く）**
  - 具体的には：研究開発 / SE / アプリケーションエンジニア / インフラ・SRE / データサイエンス / MLエンジニア / セキュリティ / IT コンサル 等を幅広くカバー
  - 企業ごとに該当職種を自動で判別し、関連する情報を優先収集
- **ガクチカは事前に聞かない**
  - ES 作成時にユーザー側で分析結果とガクチカをすり合わせる前提
  - セクション 12-3（自己 PR / ガクチカ接続）では、「求める人物像のキーワード抽出」と「過去内定者のガクチカ傾向」までを提示し、**具体的な対応表の右列は空欄（ユーザーが後で埋める）** とする
  - セクション 12-1（志望動機ドラフト）も「ガクチカと接続しやすい切り口の提示」までに留め、**個別エピソードの推測は行わない**

企業名だけ与えられれば即実行可能。不明点があっても勝手に仮定せず、調査内容をベースに進める。

### Step 1. 業界判定 → 業界別リファレンス読込
企業が属する業界を判定し、`references/industries/README.md` → 該当 1〜2 ファイルを読む。
全業界ファイルを読む必要はない。

### Step 2. 情報収集
以下を**全て**（アクセス可能な範囲で）確認する。

**一次情報（最優先）**
1. コーポレートサイト（会社概要・事業・理念・プレスリリース）
2. 採用サイト（求める人物像・職種・社員インタビュー・選考フロー）
3. **IR 資料**（詳細は `references/ir_reading.md` 参照）
   - 有価証券報告書 / 決算短信 / 決算説明資料 / 中期経営計画 / 統合報告書
4. 代表者メッセージ・CEO インタビュー

**二次情報**
5. 業界地図・四季報（業界内順位、シェア、併願先の特定に使う）
6. 日経・Reuters 等のニュース（直近 1 年）
7. OB/OG インタビュー記事

**競合・業界内ポジション（セクション 5 の一次ソース・必須）**
業界内での立ち位置を示すため、以下を体系的に収集する：
- **直接競合 2〜3 社**の IR 資料（決算説明資料・中計）と採用サイト
- **隣接競合 2〜3 社**の事業概要と主力セグメント
- **業界ベンチマーク 1〜2 社**（業界トップ / 海外勢）の概要
- 業界全体の **市場規模・成長率・上位企業シェア**（業界地図・調査会社レポート）
- 就活体験談から**併願先の傾向**を把握（「この企業を受ける人は他にどこを受けているか」）

**グループ企業の場合は追加（該当時のみ）**
- 親会社・持株会社の IR 資料（グループ全体の戦略が分かる）
- 兄弟企業（同グループ内の主要関連会社）2〜3 社の概要
- グループ全体の組織図（親会社の統合報告書・コーポレートサイトに通常掲載）
- 直近のグループ再編・統合ニュース

**就活体験談（セクション 12 の一次ソース・必須）**
8. ワンキャリア・Unistyle・外資就活・就活会議・みん就
9. note・個人ブログ・X

**情報系院生として追加**
10. 技術ブログ・GitHub・OSS・カンファレンス登壇
11. R&D・研究所の公開情報（論文・特許）

- **グループ企業の場合**：`{親会社名} グループ 組織図` / `{兄弟企業名} {企業名} 違い` / `{企業名} グループ内 位置付け`

## 全体の重要ルール

### 粒度・網羅性
- **省略しない**。埋まらないセクションも見出しは残し「要確認」と書く
- 数値は具体値（「売上 XX 億円（YYYY 年度）」）、**直近 3 年の推移**
- 数字は出典なしで書かない（「〇〇によると」を付ける）

### 選考情報の新鮮度
- 対象年度（志望年度）を必ず明記。年度が違う情報を混ぜない
- 最低 3 ソース（公式＋就活口コミ 2 種以上）でクロスチェック
- 過去情報には「毎年変わる可能性あり」注記

### 体験談の取得状況の可視化
- セクション 12 冒頭にサマリーテーブル必須
- 見つからなかった情報は明記、推論で埋めない
- アクセスできなかったソースも明記

### ユーザー個人情報の取り扱い（重要）
- **ガクチカ・研究内容・強みなどユーザー個人の情報を事前に聞かない・推測しない**
- レポートは「企業側の情報整理」に徹する。**接続はユーザー側が後工程で行う前提**
- 「あなたのエピソード」「自分の研究テーマ」等の空欄は、ユーザー記入欄として残す（勝手に埋めない）
- 志望動機ドラフトも「切り口の提示」までに留め、**完成形の文章は書かない**

### 競合比較の精度（重要）
- **比較対象は合計 5〜7 社**（直接 2〜3 + 隣接 2〜3 + ベンチマーク 1〜2）。2〜3 社だけでは浅い
- 競合も本命と**同じ粒度**で IR・採用サイトを確認する（手抜きしない）
- 比較は 4 つのレイヤーで行う：**定量 / 事業戦略 / 技術・R&D / カルチャー**
- 「なぜこの企業か」は **論点別（事業軸・技術軸・カルチャー軸）** に分解して構造化する
- **数値・固有名詞**を必ず引用（「売上 X 億円」「中計で〇〇戦略を打ち出し」等）
- 併願先は就活体験談から実データを拾う（推測ではなく）

### 差別化の視点（ユーザー最重視）
- 中期経営計画の具体的な数値目標・新規事業名を引用
- 有価証券報告書「事業等のリスク」への切り込み（他の就活生がほぼ見ない）
- 競合比較での「なぜ A 社ではなくこの企業か」を**論点別に分解**して論証
- 技術ブログ・OSS 活動・特許などのエンジニアリング文化の裏側（情報系の強み）

### 情報ソースの優先順位
1. 企業公式（IR・採用サイト・統合報告書）
2. 大手メディア（日経・Reuters・NHK）
3. 業界地図・四季報
4. 大手就活サイト（ワンキャリア・Unistyle・外資就活）
5. 口コミサイト（OpenWork 等）— 複数照合
6. 個人ブログ・SNS — 裏取り必須

### 注意点
- 口コミサイトの憶測や個人の悪口はそのまま載せない
- 「ブラック」「斜陽」等の主観的レッテルは避け、事実ベースで記述
